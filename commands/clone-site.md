# 网站 1:1 复刻任务

## Step 0 — 先确认目标，不要跳过这一步

检查用户触发这个命令时是否已经给了网址（`$ARGUMENTS`）。

**如果 `$ARGUMENTS` 是空的，或者只有网址但没有其他上下文**，你必须先向用户提出以下问题，
拿到回答之后才能继续，不要在没有明确目标的情况下开始分析：

1. "你想 1:1 复刻哪个网站？请贴上完整网址（含 https://）。"
2. "这个网站是你自己拥有 / 有明确权限操作的吗？（比如你自己的作品集、你负责的公司官网）"
   - 如果用户说是别人的网站、且目的是复刻后对外发布/商用：明确提醒——这个工具应该用在你自己
     拥有版权或已获授权的网站上；对他人受版权保护的商业网站做私下学习研究是一回事，但不应该把
     复刻结果重新对外发布或商用，可能涉及版权问题。请对方确认理解后再继续，或建议对方只做局部
     UI 手法学习而非整站复刻。
3. （可选追问，用户没主动说明时问一句）"这次复刻，桌面端和移动端都要覆盖吗？有没有特别想还原
   的某个页面区块或动效？"

如果用户已经在触发消息里给了网址并说明了这是自己的网站，跳过第 1、2 题，直接进入 Step 1。

## Step 1 — 全局探索与取证

对目标网站做多视口、多状态截图（桌面 + 移动、首屏 + 各主要滚动区段、暗态/亮态如果支持切换）：

- **优先**使用已连接的 Playwright MCP 或类似浏览器自动化工具。
- **没有的话**，退化使用当前环境自带的浏览工具完成（例如 Cowork 里的 Claude in Chrome / computer
  use，或任何已连接的浏览器能力）。
- 如果以上都不可用，请用户自己提供截图，并明确告诉用户需要哪些视角。

同时盘点页面用到的模块、资源、第三方库，落盘到 `docs/module-inventory.md`。

## Step 2 — 提取 Design DNA

把网站 URL 与上一步的截图交给 **design-dna** 技能（Analyze 阶段），产出字段齐全、无量化的
`docs/design-dna.json`。之后任何模块的颜色/字号/间距/缓动/断点，一律以这份 JSON 为准，不得凭
感觉现场定。

## Step 3 — 动效模块逆向

对每一个含特效的模块（WebGL/Canvas/shader/滚动驱动动效等超出普通 CSS 的部分），用
**web-shader-extractor** 技能产出 `docs/effects/<模块>.md`（含着色器/资源/渲染图/时序/输入证据
+ 本地可跑基线）。

## Step 4 — 模块化重建

按 `docs/module-inventory.md` 的依赖顺序逐个实现模块。每个模块开工前，先用 **duck** 技能把
"这个模块的数据流/状态/动效触发条件"像对新手一样逐层讲一遍，暴露隐藏假设，确认没有遗漏再动手写。

实现时严格对照 `docs/design-dna.json` 取值、对照 `docs/effects/<模块>.md` 还原动效。

## Step 5 — 闭环验证

- 自己运行项目，在与原站相同的视口/滚动位置截图，并排比对原站截图。
- 以 `docs/design-dna.json` 为量化判据逐项核对：色彩、字阶、间距、层级、动效曲线、特效清单。
- 以 `docs/effects/<模块>.md` 为动效判据核对：粒子/着色器/滚动行为是否与原站一致。
- 需要精修时，触发 design-dna 技能的"精修轮次"：把同一批原站截图/URL 再次审视，对照参考复审
  界面层级与点缀、字阶与留白、动效与材质及整体 UI，把结论回填实现，不需要从零重做。
- 把每轮比对结果与差距记录写进 `docs/verify/<模块>.md`，迭代至与线上生产环境视觉与动效一致。

## 落盘原则

每个模块的分析与实现结论都写进本地 `docs/`，即便上下文被压缩、会话被中断，后续也能凭本地文档
无损续接，不需要用户重新讲一遍需求。
